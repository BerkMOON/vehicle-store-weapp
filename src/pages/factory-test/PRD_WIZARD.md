## 1. 背景与目标

### 1.1 背景

- 现有入场检测页将 VIN 查询、绑定提示、固件/失效、里程、工单、碰撞上报集中在一屏展示。
- 车间师傅更适应**一次只看一件事、点下一步继续**的操作习惯。

### 1.2 目标

| 目标 | 说明 |
|------|------|
| 降低认知负担 | 每步仅展示当前动作所需信息 |
| 规范操作顺序 | 车架号 → 设备处置 → 检测/修复 → 里程 → 事故 → 完成 |
| 可考核可追溯 | 每次拍/录 VIN 及完整流程需落库，供门店当日进厂比对、考核是否执行 |

### 1.3 非目标（本期不做）

- 固件 OTA 一键升级（仍为线下手工升级 + 文案指引）
- 碰撞业绩/现金发放
- 在小程序内区分展示「失效」与「旧版本」明细（合并为「设备需修复」）

## 2. 用户与场景

- **用户**：门店设备工程师 / 车间师傅  
- **场景**：车辆进厂后，按标准流程完成入场检测并留痕

---

## 3. 流程总览（向导步骤）

```text
[1 拍车架号] → [2 设备情况] → [3 补绑定?] → [4 设备检测] → [5 里程更新] → [6 相关事故] → [7 完成]
                  ↓ 无易达安              ↓ 需修复              ↓ 未检出上报
                  直接完成               修复教程→继续          可选弹窗
```

| 步骤 | 名称 | 说明 |
|------|------|------|
| 1 | 拍车架号 | 拍照 OCR 或手输 VIN，确认后查询设备；**记录 VIN 扫描** |
| 2 | 设备情况 | 展示是否绑定；二选一：**① 没有易达安设备** **② 有，补绑定**（已绑定则直接进入步骤 4） |
| 3 | 补绑定 | 仅「有设备未绑定」时；录入 SN 绑定后刷新 |
| 4 | 设备检测 | 后台检测：失效 **或** 非最新版本 → 统一提示 **「设备需修复」** + 修复教程（小曹提供，本期占位）→「已修复，下一步」 |
| 5 | 里程更新 | 工程师录入仪表盘里程，提交后进入下一步 |
| 6 | 相关事故 | 仅展示 **待认领**、**待返厂** 工单；支持进详情操作；**未检出事故上报**（碰撞线索） |
| 7 | 完成 | 展示「检测完成 100%」；**上报完整流程记录**；可「再检一台」 |

---

## 4. 步骤详细说明

### 4.1 步骤 1：拍车架号

- 支持：拍照识别（行驶证 / 铭牌）、手动输入。
- 主按钮：**确认并查询**（校验 VIN 格式可选，与现网一致）。
- 成功后：调用 `DeviceAPI.list({ vin })`；调用 `recordVinScan` 上报当日车架号留痕。

### 4.2 步骤 2：设备情况

**展示：**

- 当前 VIN（只读）
- 车架号绑定状态：以设备记录上的 **VIN 是否与当前车架号一致** 为准（**不用** `status` 字段；该字段表示店端设备状态/安装上报，与用户微信绑定无关，工程师无法操作）

**操作：**

| 选项 | 条件 | 下一步 |
|------|------|--------|
| ① 没有易达安设备 | 任意 | 跳过绑定/检测/里程/事故 → **步骤 7 完成**（`device_path=no_device`） |
| ② 有，补绑车架号 | 列表中无与当前 VIN 一致的设备记录 | **步骤 3**（录入 SN，调用 `device/update` 写入 VIN） |
| 继续检测 | 已存在 VIN 匹配的设备记录 | **步骤 4** |

### 4.3 步骤 3：补绑定

- 输入设备 SN（必填），可选手机号等（与现 `EntryCheckAPI.bindDevice` 一致）。
- 绑定成功 → 拉取 `offlineVersionSummary` → **步骤 4**。

### 4.4 步骤 4：设备检测（合并失效 + 版本）

**检测逻辑（前端，与现网数据同源）：**

- `offlineVersionSummary.offline_over_n_days === true` → 需修复  
- 或 当前固件/算法版本 ≠ OTA 最新 → 需修复  
- 否则 → 显示「设备正常」，可直接下一步  

**需修复时 UI：**

- 文案：**设备需修复**（不分别写失效/旧版本）
- 修复教程区：占位「教程由小曹提供，敬请期待」+ 简要线下操作提示
- 按钮：**已修复，继续**（师傅确认后进入里程；记录 `repair_ack`）

### 4.5 步骤 5：里程更新

- 不展示系统上报里程。
- 录入最新里程 → `DeviceAPI.updateMileage`（或后续 `entry-check/mileage`）。
- 按钮文案：**更新完成，下一步**。

### 4.6 步骤 6：相关事故

**工单列表过滤：**

- 仅 `status` 为 **待认领**（`pending`）、**待返厂**（`waitingForReturn`）。
- 不展示：已处理、已完成、已拒绝、战败等（与 `TaskAPI.List` + `vin` 筛选联调）。

**操作：**

- 列表项可跳转工单详情（现有 `order-detail`）。
- **未检出事故上报**：打开碰撞上报弹窗（简化字段，无失效分支文案）；上报成功或用户确认「本车无事故」均可进入完成页。
- **无需上报，完成检测**：无事故时直接进入步骤 7。

### 4.7 步骤 7：完成

- 展示：**入场检测完成 100%**（进度环或百分比文案）。
- 汇总：VIN、设备路径、是否需修复、里程是否已更新、是否上报碰撞。
- 按钮：**再检一台**（重置向导）。

## 5. 当日车架号留痕（考核）

### 5.1 业务要求

- 步骤 1 **确认车架号** 时上报，仅存储当日扫描的车架号。
- **同一公司 + 门店 + 车架号 + 自然日**：再次上传则覆盖（更新工程师 id、ctime）。
- **不同自然日**：写入新记录。

### 5.2 接口

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/business/entry-check/inspection/record` | body：`{ "vin": "..." }` |
| GET | `/api/admin/device/entry-inspection-log/list` | 查询：公司、门店、车架号、日期 |

表字段：`company_id`, `store_id`, `business_user_id`, `vin`, `ctime`, `mtime`

### 5.3 前端

- 步骤 1 调用 `recordVinScan`；完成页不再上报。

## 7. 验收标准

1. 任一步骤全屏仅展示当前步骤内容，顶部有步骤进度（如 2/6）。
2. 「没有易达安」路径不出现绑定、设备检测步骤。
3. 「需修复」不出现「失效」「旧版本」分栏，仅统一「设备需修复」。
4. 工单列表仅待认领、待返厂。
5. 完成页显示 100% 完成态，可再检一台。
6. 步骤 1 确认车架号后产生一条当日留痕（同日同 VIN 覆盖）。
