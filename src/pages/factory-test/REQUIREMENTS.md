# 入场检测页（factory-test）需求说明与接口对照

本文档根据业务口述整理，描述页面改版后的流程、异常分支，并对照当前前端仓库内已有请求封装与建议新增接口。

---

## 一、页面定位

门店工程师对到店车辆进行「入场检测」：以**车架号（VIN）**为入口，串联「是否有设备 → 是否绑定 → 固件是否最新 → **是否失效设备（可主动查询，仅反映当前状态）** → **里程校对（录入）** → 工单是否可查可操作 → **碰撞线索上报（一次提交；是否失效仅按当前能力）**」整条链路。

---

## 二、主流程（建议交互顺序）

1. **录入车架号**
  - 方式 A：员工手动输入（需格式校验，常见 17 位 VIN 规则可按产品约定）。
  - 方式 B：**微信小程序车架号 OCR 插件**拍照识别（需在小程序后台配置插件、在 `app.config` 中声明 `plugins`）；识别结果允许人工校对后再提交。
2. **按车架号查询设备**
  - 调用后端能力：根据 VIN 返回当前门店/账号可见范围内的设备信息（含绑定状态、SN 等）。接口若返回系统上报里程，仅作后台或其它模块使用；**本页面向师傅不做系统里程展示**（见第 6 条）。
3. **分支判断**
  - **无任何设备记录**：明确提示「未查询到设备」，引导确认用户是否本就没有安装记录仪；若确认未安装，流程结束或记录检测结果。
  - **现场已安装但系统查不到**：判定为**未绑定**，引导执行**绑定**动作（绑定成功后刷新查询）。
  - **已安装且已绑定**：继续后续检测项。
4. **固件 / 设备版本**
  - 展示设备当前版本号；与**服务端定义的最新版本**比对。
  - 若非最新：页面仅做**提示与操作说明**（如步骤文案、注意事项）；**固件更新由工程师在线下对设备手动完成**，小程序内不做一键升级、也不依赖应用内 OTA 流程。
5. **失效设备查询（独立能力）**
  - 在已绑定设备、可获取 `sn` / `device_id` 的前提下，支持**主动查询当前设备是否属于失效设备**（如低于业务阈值的固件/算法版本、列入淘汰名单的机型等，**以后端规则为准**）。
  - **能力边界**：仅能判断**当前时刻**设备是否处于失效状态；**无法**依据事故时间推断「在事故发生时是否属于失效设备」。查询结果用于**入场检测页展示与风险提示**（如「当前为失效设备」）；与第 8 条「上报碰撞」**不做**「事故时间 + 失效」的二次判定联动。
  - 不要求用户先填事故时间即可完成「是否失效」的查询，便于师傅提前知情。
6. **里程校对（录入）**
  - **不向师傅展示系统侧上报里程**（系统里程与实表必有误差，容易造成误导）。
  - 页面提供「里程校对」：**工程师输入当前最新里程**（以仪表盘/实车为准），并**记录校对提交时间**（服务端落库，便于追溯）。
7. **工单情况**
  - 展示与该用户/车辆相关的工单列表（状态、时间等）。
  - 工程师可进行现有能力范围内的操作（如认领、跟进等，与工单详情页能力对齐）。
8. **碰撞线索上报（与业绩关联，发放不在小程序）**
  - 提供按钮 **「上报碰撞」**。
  - 工程师填写 **事故时间、工程师姓名**；按需上传事故整体照片等（字段以后端为准）。
  - **不做「事故时间范围内的失效判定」**：系统**不能**根据事故时间判断当时是否为失效设备，**只能**知道**当前**是否为失效设备（见第 5 条主动查询）。因此**上报时直接一次提交即可**，无需在流程中增加「填事故时间 → 再调 invalid-check → 再分支是否拍照」的步骤；若需落库「当前是否失效」等标签，由**上报接口一次写入**（前端可带入第 5 条已查结果，或由后端按当前规则在写入时计算），产品文案与必填项（如是否必须照片）由业务与后端统一约定。
  - 业务侧据此统计业绩、发现金奖励等；**奖励发放功能不在小程序内实现**，小程序仅完成信息采集与上报。

## 三、分支逻辑小结


| 场景    | 表现                | 期望动作                                                                    |
| ----- | ----------------- | ----------------------------------------------------------------------- |
| 查无设备  | 接口返回空列表           | 提示无设备；可选记录检测结果                                                          |
| 有车无绑定 | 现场有设备但查询无绑定关系     | 提示未绑定 → 走绑定流程 → 再查询                                                     |
| 已绑定   | 有设备且 status 等为已绑定 | 版本检查 → **失效设备查询（可选，仅当前状态）** → **里程校对录入** → 工单 → **上报碰撞（一次提交；不按事故时间做失效判定）** |

## 四、前端仓库内「已有」接口封装（可直接或稍加参数即用）

以下路径均为 `src/request` 下当前存在的封装（具体 URL、字段以后端为准）。


| 能力              | 封装                                     | 说明                                                                                                                                         |
| --------------- | -------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------ |
| 按条件查设备列表（含 VIN） | `DeviceAPI.list` + `DeviceRequest.vin` | `GET /api/business/device/list`，请求体支持 `vin`、`sn`、`status`（文档注释：`init` 未绑定，`bound` 已绑定）等；返回 `device_list` 含 `vin`、`sn`、`mileage`、`status` 等 |
| 设备统计（与本页弱相关）    | `DeviceAPI.stat`                       | 门店维度统计，入场检测可选展示                                                                                                                            |
| 流失/丢失类列表查询      | `DeviceAPI.getLossNotifications`       | `GET .../getLossNotifications`，参数含 `sn`、`device_id`、时间范围等；**当前为查询列表**，非「提交丢失线索」                                                            |
| 里程提醒列表          | `DeviceAPI.getMileageList`             | 偏列表运营场景；**入场检测页的里程校对不以列表接口为准**，不向师傅展示系统里程                                                                                                  |
| 工单列表            | `TaskAPI.List`                         | `GET /api/business/task/list`；当前 TypeScript 类型里仅有 `offset`、`limit`、`status`，**未见 `vin` / `phone` / `sn` 筛选字段**                             |
| 工单认领            | `TaskAPI.Accept`                       | 已有                                                                                                                                         |
| 工单跟进            | `TaskAPI.Process`                      | 已有                                                                                                                                         |
| 工单详情            | `TaskAPI.Detail(clueId)`               | 已有                                                                                                                                         |
| 出厂检测占位          | `FactoryTestAPI.*`                     | 前端占位，与新版入场检测流程关系不大，可合并或废弃                                                                                                                  |

## 五、建议「新建」或由后端扩展的接口 / 能力


| 需求点               | 现状                                       | 建议                                                                                                                                                                                                                                                                                                                                |
| ----------------- | ---------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 车架号 OCR           | 仓库内暂无封装                                  | **采用微信小程序官方/业务已接入的车架号 OCR 插件**完成拍照识别；需在工程中接入插件 API，无需自建后端识别作为主路径                                                                                                                                                                                                                                                                  |
| 设备绑定              | `deviceApi` 无 bind 类方法                   | 新增例如 `POST /api/business/device/bind`（参数：`vin`、`sn`、门店等，以后端为准）                                                                                                                                                                                                                                                                    |
| 最新固件版本 / 是否需升级    | 请求层无 version/firmware 字段                 | 后端返回**当前版本号 + 最新版本号（及比对结果）即可；前端展示差异与手工升级指引**。升级动作为工程师线下操作，不要求小程序内触发升级接口                                                                                                                                                                                                                                                           |
| 工单按 VIN（或用户）筛选    | `ListRequest` 类型未声明 vin                  | **后端扩展** `task/list` 查询参数（如 `vin`、`sn`），并同步更新 `taskApi/typings`；或新增 `GET /api/business/task/listByVin`                                                                                                                                                                                                                            |
| 里程校对提交            | 无专门接口                                    | 新增 `POST .../entry-check/mileage`（或合并入场检测）：**工程师录入里程数值 + 校对提交时间**（必选）；无需上传「与系统里程比对」类字段                                                                                                                                                                                                                                            |
| 碰撞线索上报            | 仅有 `getLossNotifications` 查询             | 新增 **碰撞/线索上报** 接口：`POST .../entry-check/collision-report`；**图片上传**可单独 `POST .../entry-check/collision-upload`（小程序 `uploadFile`）；后端承接业绩统计，**不包含奖励发放**                                                                                                                                                                              |
| **查询是否为失效设备（通用）** | 无独立「仅查失效」接口；历史上曾设想碰撞内 `invalid-check`（带事故时间） | 新增 `GET` 或 `POST .../entry-check/invalid-device`（或 `.../device/invalid-status`）：入参 **`vin` + `sn`（或 `device_id`）**，**不要求事故时间**；出参 **`is_invalid_device: boolean`**，可选 **`reason`**、**`rule_version`**。仅表达**当前**是否失效，供入场检测页展示；**不与事故时间联动**。 |
| 碰撞流程内「按事故时间的失效判定」 | 产品/能力上不可行                         | **不依赖**：无法在事故时间维度判定失效，**不要求**单独 `POST .../entry-check/invalid-check`（`accident_time`）作为上报前置步骤；碰撞上报以 **`POST .../entry-check/collision-report`** 一次提交为主。 |
| 入场检测一站式聚合（可选）     | 无                                        | 为减少多次请求，可由后端提供 `GET .../entry-check/summary?vin=` 一次返回：设备、绑定状态、版本比对结果、**是否失效**、工单摘要等（**可不返回展示给师傅的系统里程**，或与校对分离）                                                                                                                                                                                                                   |

## 六、与现有数据字段的对应关系（便于联调）

- **设备是否绑定**：可结合 `DeviceAPI.list` 返回条目的 `status`（及是否有记录）；若「有车未绑定」需单独状态码或字段，需后端约定。
- **里程**：入场检测页**不展示** `DeviceList.mileage` 等系统上报值；仅提交工程师录入的校对里程与校对时间。
- **工单**：`TaskInfo` 含 `vin`、`sn`、`clue_id` 等；列表若不支持按 VIN 过滤，前端只能拉全量再筛（不推荐），应走后端筛选。
- **碰撞上报**：与流失提醒列表可能相关但入口不同；提交字段含事故时间、工程师姓名、照片等（是否必填以后端为准）；**不按事故时间做失效分支**；业绩与现金奖励由后台/other 系统处理。
- **失效设备**：仅**当前状态**可查询（`invalid-device` 等）；与碰撞上报**解耦**；前端如需可封装 `queryInvalidDevice`；**不必**再封装「带事故时间的 `checkInvalidDevice`」作为上报必经步骤。

## 七、后续前端改造提示（非本文档范围，仅作备忘）

- 将当前 factory-test 页从「SN + 模拟硬件自检」切换为「VIN 驱动 + 上述业务流程」。
- 车架号拍照识别：**接入小程序 OCR 插件**（按插件文档调起识别，回填 VIN）；手动输入与 OCR 结果均需可做格式校验。
- 里程：仅表单项「录入最新里程」+ 提交时带**校对时间**；不展示 `DeviceList.mileage`。
- **碰撞**：**无法**在事故时间维度判断当时是否失效，**只能**知道当前是否失效；**上报时直接一次提交**（事故时间、工程师姓名、照片等），**不要**设计「先填事故时间再调带 `accident_time` 的 invalid-check、再分支是否拍照」的流程；若需展示「当前失效」提示，仅复用第 5 条**当前态**查询结果或由上报告知后端自行落库。
- **失效设备**：绑定成功后增加「查询是否失效设备」按钮或自动请求，展示**当前**是否失效及文案；与碰撞上报并行参考即可，**不要求**与事故时间、上报步骤强绑定。
- 工单操作：可复用订单列表与详情页的组件或跳转至 `order-detail`。

## 八、修订记录

- 初版：根据产品口述整理需求与接口对照（前端仓库检索日期以当前代码为准）。
- 修订：车架号 OCR 明确为**微信小程序插件**方案；固件更新明确为**工程师手动操作**，小程序仅提示与说明。
- 修订：**里程**改为仅录入校对里程 + 记录校对时间，**不展示系统里程**；**碰撞**曾描述为「事故时间 + 失效分支 + 非失效拍照」；业绩与奖励**发放不在小程序**。
- 修订：新增**失效设备通用查询**（不依赖事故时间）；曾规划与碰撞内 **`invalid-check`（带事故时间）** 联动，后因能力边界（**无法**在事故时间维度判定失效）改为以 **`collision-report` 一次提交**为主，**不再**以带事故时间的 invalid-check 为必经步骤（见末条修订）。
- 修订：**第七条（本节）及全文对齐**：明确**不能**在事故时间范围内判定是否失效，**仅能**判断**当前**是否失效；碰撞上报改为**直接一次提交**，不再把「带事故时间的 invalid-check」作为产品必经步骤；第五节、第六节、第二节第 8 条同步修改。
